### 问题清单

| 编号 | 优先级 | 问题描述 | 建议修复方案 |
|------|--------|----------|------------|
| 001 | P0 | **Celery 动态限流实现无效**：`batch_compile_l0` 中通过 `compile_tenant_task.rate_limit = '100/m'` 动态设置限流不可靠。Celery 限流通常在 Worker 端配置，客户端动态修改存在竞态条件，且多 Worker 环境下是单节点限流而非全局限流，无法有效防止 L0 变更引发的编译风暴。 | 1. 移除客户端动态限流代码。<br>2. 在 Celery Worker 启动配置中设置全局 `task_rate_limit`。<br>3. 或使用 Celery Canvas 的 `chord` 配合自定义 throttling 中间件，或在 API 层实现令牌桶限流后再提交任务。 |
| 002 | P0 | **编译看门狗存在竞态风险**：`_maintenance_window_compile` 设置 5 分钟 TTL 状态，但 `compile_task` 成功后未明确清除 `L0_HIGH_COMPILING` 状态。若编译耗时超过 5 分钟，看门狗会将状态回退为 `STALE`，而随后完成的编译任务可能无法正确更新状态，导致系统状态不一致或查询被错误阻断。 | 1. 在 `compile_task` 成功执行后，显式调用缓存服务清除 `COMPILING` 状态并设置为 `READY`。<br>2. 看门狗任务仅在状态仍为 `COMPILING` 且超时时才回退，需确保成功回调优先于看门狗执行（如延长看门狗倒计时或使用分布式锁）。 |
| 003 | P1 | **SQL 标识符引用安全性不足**：`_quote_identifier` 使用手动字符串拼接 (`f'"{escaped}"'`) 而非数据库驱动提供的安全方法。虽然前有白名单校验，但手动实现易遗漏边缘字符处理，且 `_validate_sql_safety` 依赖黑名单正则，易被绕过。 | 1. 推荐使用 `psycopg3` 的异步连接池，其支持 `cursor.execute(query, params)` 自动处理标识符安全（或通过 `sql.Identifier`）。<br>2. 若必须手动处理，需引入更严格的白名单校验逻辑，移除黑名单校验依赖。 |
| 004 | P1 | **多进程模型加载内存风险**：`IntentClassifier` 使用单例模式，但在 Gunicorn/Uvicorn 多进程模式下，每个 Worker 进程会独立加载一次模型（内存不共享）。设计文档虽提及内存评估，但未规定 `preload_app` 配置，可能导致内存超标。 | 1. 明确部署文档要求开启 Gunicorn `--preload` 参数以共享模型内存。<br>2. 或改为独立模型服务（TorchServe），通过 gRPC/HTTP 调用，解耦内存依赖。 |
| 005 | P2 | **可观测性设计缺失**：文档未包含监控指标（Metrics）、日志规范（Logging）和链路追踪（Tracing）设计。无法监控编译耗时、查询延迟、缓存命中率、LLM 调用成本及错误率。 | 1. 集成 Prometheus + Grafana，定义关键业务指标（如 `compile_duration_seconds`, `query_latency_ms`）。<br>2. 集成 OpenTelemetry 实现全链路追踪。<br>3. 规范日志格式（JSON），包含 TraceID。 |
| 006 | P2 | **L0 编译任务队列积压风险**：`batch_compile_l0` 一次性向 Celery Broker 提交大量任务（即使分块），若租户量大（如 10 万+），可能压垮 Redis/RabbitMQ 消息队列，导致其他任务饥饿。 | 1. 引入任务提交层的背压机制（Backpressure），当队列长度超过阈值时暂停提交。<br>2. 使用独立的 Celery Queue 专门处理编译任务，与查询任务隔离。 |
| 007 | P2 | **LLM 成本与超时控制缺失**：`_llm_parse` 作为 Fallback 机制，未设置 Token 消耗预算、超时控制及熔断策略。高频低置信度查询可能导致 LLM 费用激增或响应超时阻塞线程。 | 1. 增加 LLM 调用频率限制（ per tenant/user）。<br>2. 设置严格的 `timeout` 和 `max_tokens`。<br>3. 实现 LLM 服务熔断器，连续失败时暂时降级为“无法理解”。 |
| 008 | P3 | **数据迁移方案缺失**：文档未提及从 v1.0 到 v2.0 的数据迁移策略（GraphDB Schema 变更、PG 规则表结构变更）。 | 补充数据迁移章节，包含停机窗口、回滚方案及数据校验脚本。 |
| 009 | P3 | **异常处理粒度不足**：`compile_task` 中 `except Exception` 捕获所有异常并重试，可能掩盖永久性错误（如规则语法错误），导致无效重试浪费资源。 | 区分 transient error（网络抖动）和 permanent error（规则错误）。永久错误应标记任务失败并发送告警，而非重试。 |
| 010 | P4 | **硬编码配置较多**：代码中存在硬编码值（如 `chunk_size = 1000`, `compile_timeout = 300`），不利于运维调整。 | 将配置项提取至配置中心（如 Apollo/Nacos）或环境变量，支持动态调整。 |

### 评审结论：有条件通过

**理由：**
1.  **核心稳定性风险（P0）**：Celery 限流实现逻辑存在缺陷，无法有效防护 L0 变更引发的编译风暴，可能导致数据库或服务不可用；编译看门狗逻辑存在状态不一致风险，影响业务连续性。这两点必须在开发前修复。
2.  **安全性隐患（P1）**：SQL 生成层的标识符处理未采用最佳实践，存在潜在注入风险；多进程模型加载未明确内存隔离策略，可能引发 OOM。
3.  **运维支撑不足（P2）**：缺乏监控和可观测性设计，系统上线后将处于“黑盒”状态，难以排查问题。
4.  **架构合理性**：整体分层架构清晰，读写分离、缓存策略、增量编译思路正确，符合高性能系统设计原则。

**后续行动建议：**
1.  **立即修复 P0 问题**：重新设计 Celery 限流方案及编译状态机逻辑。
2.  **补充设计文档**：增加监控指标定义、部署架构细节（含 preload 配置）、数据迁移方案。
3.  **安全加固**：审查 SQL 生成代码，优先使用驱动层安全特性。
4.  **POC 验证**：针对 L0 批量编译场景进行压力测试，验证限流和队列背压机制的有效性。
