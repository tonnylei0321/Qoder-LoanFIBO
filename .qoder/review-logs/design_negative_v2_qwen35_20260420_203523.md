### 问题清单

| 编号 | 优先级 | 问题描述 | 建议修复方案 |
|------|--------|----------|------------|
| 001 | P0 | **Celery 异步任务实现错误**：代码中 `@celery_app.task` 装饰器直接修饰 `async def` 函数。标准 Celery _worker_ 默认不支持异步函数，除非配置特定事件循环（如 gevent/eventlet）或使用 Celery 5.3+ 的特定异步支持配置。当前写法会导致任务阻塞或运行时错误。 | 1. 明确 Celery _worker_ 启动参数（如 `-P gevent`）；<br>2. 或在 Task 内部使用 `asyncio.run()` 包裹异步逻辑；<br>3. 或重构为同步 IO 操作以避免异步复杂性。 |
| 002 | P1 | **NLQ 架构设计与代码实现矛盾**：3.1 架构图明确包含"LLM 解析分支（置信度<=0.9）"，但 3.2 `IntentClassifier` 代码中完全缺失 LLM 调用逻辑，仅实现了 BERT 和模板匹配。这将导致低置信度查询无法处理。 | 1. 补充 LLM 调用接口设计（如 LangChain 集成）；<br>2. 在 `classify` 方法中增加 LLM  fallback 逻辑；<br>3. 更新代码以匹配架构设计。 |
| 003 | P1 | **SQL 注入安全风险**：`SQLGenerator` 和 `DSLExecutor` 负责将自然语言/DSL 转为 SQL，但设计中未明确参数化查询（Parameterized Query）或白名单校验机制。若直接拼接字符串，存在极高 SQL 注入风险。 | 1. 强制使用 ORM 或预编译语句（Prepared Statements）；<br>2. 对表名、字段名实施严格的白名单校验；<br>3. 在安全设计章节补充防注入策略。 |
| 004 | P1 | **本地执行模式安全隐患**：`DSLExecutor` 中若 `use_wasm=False`，则 fallback 到 `_execute_local`。若该函数内部使用 `eval` 或类似动态执行机制，将导致远程代码执行（RCE）风险。 | 1. 禁用本地动态执行模式，强制使用 WASM 沙箱；<br>2. 若必须保留，需实现严格的 AST 白名单校验，禁止任意函数调用。 |
| 005 | P2 | **L0 规则变更引发“编译风暴”**：设计指出 L0 变更需“全量编译所有租户”。若租户量大，瞬时触发数千编译任务将打爆 Celery 队列和数据库连接，导致服务不可用。 | 1. 引入编译任务限流（Rate Limiting）；<br>2. 实现灰度发布机制，分批触发租户编译；<br>3. 增加 L0 规则变更的预演环境。 |
| 006 | P2 | **多源数据一致性风险**：规则分散在 GraphDB (L0/L1) 和 PostgreSQL (L2)。编译时合并逻辑 `_merge_rules` 未定义冲突解决策略（如 L2 是否强制覆盖 L1）。若数据源更新不同步，可能导致规则执行不一致。 | 1. 明确规则优先级覆盖策略（L2 > L1 > L0）；<br>2. 引入分布式事务或最终一致性校验机制；<br>3. 增加规则合并的单元测试覆盖。 |
| 007 | P2 | **GraphDB 查询性能未优化**：L0/L1 规则存储在 GraphDB，高频读取若仅依赖缓存，一旦缓存失效，复杂的 RDF 查询可能导致高延迟。设计中缺乏索引或物化视图策略。 | 1. 设计 GraphDB 专用索引（如 Lucene 索引）；<br>2. 考虑将热点规则同步到 Redis 或 PG 以减少 GraphDB 依赖；<br>3. 补充 GraphDB 查询性能压测计划。 |
| 008 | P3 | **Docker Compose 配置无效**：7.1 节中 `replicas: 3` 在标准 Docker Compose 中无效，需配合 Swarm Mode 或使用 Kubernetes。当前配置无法实现多副本部署。 | 1. 若使用 K8s，提供 Deployment YAML；<br>2. 若坚持 Compose，移除 replicas 并使用 `docker-compose up --scale`。 |

| 010 | P3 | **BERT 延迟假设过于乐观**：3.2 节假设 BERT 分类延迟 `<5ms`。在 CPU 环境下，BERT-base 推理通常需 20-50ms。若硬件未指定，可能达不到 SLA。 | 1. 明确推理硬件要求（如 GPU 或特定 CPU 指令集）；<br>2. 引入模型量化（Quantization）优化；<br>3. 调整 SLA 预期或增加缓存命中率要求。 |

### 评审结论：不通过

**理由：**
1.  **存在阻塞性技术缺陷 (P0)**：Celery 异步任务实现方式在当前描述下不可执行，直接阻碍开发落地。
2.  **核心安全缺失 (P1)**：SQL 生成和本地执行模式缺乏明确的安全防护设计，存在注入和 RCE 高风险，不可上线。
3.  **设计一致性不足 (P1)**：架构图与代码实现存在明显矛盾（LLM 分支缺失），显示设计未闭环。
4.  ** scalability 风险 (P2)**：缺乏对 L0 变更引发系统雪崩的防护机制。

**后续建议：**
请优先修复 P0 和 P1 级别问题，特别是**安全架构**和**任务执行模型**。修订后需重新进行安全专项评审和架构可行性验证，方可进入开发阶段。
