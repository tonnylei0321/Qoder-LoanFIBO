### 问题清单

| 编号 | 优先级 | 问题描述 | 建议修复方案 |
|------|--------|----------|------------|
| 001 | **P0** | **SQL 注入高危风险**：`SQLGenerator._build_parametrized_conditions` 中，`slot_name` 直接通过 f-string 拼接到 SQL 语句中 (`f"{slot_name} = %s"`)。虽然对 `slot_name` 做了正则校验，但未经过**数据库 schema 白名单**验证。若攻击者通过对抗样本操控意图分类器输出恶意字段名（如 `loan_amount; drop table` 的变体），正则可能绕过，导致注入。 | 1. 移除 SQL 字符串拼接，所有字段名必须从 `SemanticMapping` 允许的白色列表中查找。<br>2. 若字段名动态，需使用数据库驱动支持的 identifier quote 机制（如 PostgreSQL 的 `quote_ident`），严禁直接拼接。 |
| 002 | **P0** | **代码定义冲突**：文档中 `RuleCompiler` 类被定义了两次。第一次在 2.2 节开头（纯异步），第二次在 2.2 节后半部分（同步/异步双模式）。这导致设计文档逻辑自相矛盾，无法直接指导开发。 | 统一类定义。建议保留“同步核心 + 异步包装”的模式（第二次定义），删除第一次定义，并确保文档中所有调用方引用一致。 |
| 003 | **P1** | **WASM 编译可行性风险**：`DSLExecutor` 设计要求将 DSL 实时编译为 WASM (`_compile_to_wasm`)。自定义 DSL 到 WASM 的编译链开发成本极高，且运行时编译开销大，可能无法满足查询延迟要求。 | 1. 重新评估安全执行方案。建议使用成熟的沙箱解释器（如 `restrictedpython`）或预编译字节码。<br>2. 若坚持 WASM，需明确编译工具链（如 AssemblyScript 或 TinyGo）及预编译策略，而非运行时编译。 |
| 004 | **P1** | **Celery 与 Async 混用架构缺陷**：`IncrementalCompiler.batch_compile_l0` 中使用 `await asyncio.sleep` 和 `await self.compiler.compile`。若此方法在 Celery Worker (prefork 模式) 中运行，`asyncio` 将无法正常工作；若在 API 层运行，则阻塞 API 进程执行耗时编译任务，违背使用 Celery 的初衷。 | 1. 明确边界：API 层仅负责 `delay` 任务，不 `await` 编译结果。<br>2. Celery Worker 内严禁使用 `asyncio` 逻辑，除非配置了 `gevent`/`eventlet` 池，但文档明确声明使用 `prefork`。需改为纯同步阻塞或回调机制。 |
| 005 | **P1** | **模型加载内存爆炸风险**：`IntentClassifier` 在 `__init__` 中加载 BERT 模型。若 FastAPI 启动多 Worker（如 Uvicorn 4 进程），每个进程都会加载一次模型（440MB+），导致内存浪费且启动慢。 | 1. 使用单例模式或全局变量确保模型每进程只加载一次。<br>2. 考虑将模型服务独立部署（如 TorchServe），通过 gRPC 调用，实现资源隔离和弹性伸缩。 |
| 006 | **P2** | **缓存一致性隐患**：编译结果同时写入 Redis 和 PostgreSQL（回滚服务中提到）。在并发场景下，若 Redis 更新成功但 PG 失败（或反之），会导致查询引擎读到新规则但审计/回滚系统记录旧状态，数据不一致。 | 1. 明确 Redis 仅为缓存，PG 为唯一真理源（Source of Truth）。<br>2. 引入分布式锁（Redis Lock）保护编译/回滚过程。<br>3. 增加缓存校验机制，查询时比对 Redis 版本与 PG 版本。 |
| 007 | **P2** | **LLM 降级缺乏熔断保护**：`IntentClassifier` 在低置信度时调用 LLM。未设计 LLM 服务的超时、重试及熔断机制。若 LLM 服务抖动，将拖垮整个查询接口。 | 1. 增加 LLM 调用的 Circuit Breaker（熔断器）。<br>2. 设置严格的 Timeout（如 2 秒），超时直接返回“无法理解”而非挂起。<br>3. 增加 LLM 调用频率限制，防止成本失控。 |

### 评审结论：不通过

**理由：**
1.  **安全性致命缺陷**：存在明确的 SQL 注入风险（问题 001），在金融/规则引擎场景下不可接受。
2.  **架构逻辑矛盾**：核心类 `RuleCompiler` 定义冲突（问题 002），且异步/同步模型混用（问题 004）将导致运行时错误。
3.  **技术可行性存疑**：DSL 实时编译 WASM（问题 003）工程风险过高，缺乏明确落地路径。
4.  **性能隐患**：模型加载策略（问题 005）缺乏生产环境考量，易导致 OOM。

**建议：**
请优先修复 **P0 级安全问题** 和 **P1 级架构矛盾**。在重新提交评审前，建议进行以下工作：
1.  统一代码设计，消除类定义冲突。
2.  完善 SQL 生成器的白名单校验逻辑。
3.  重新评估 DSL 执行引擎的技术选型。
4.  补充完整的回滚服务代码及可观测性设计。
