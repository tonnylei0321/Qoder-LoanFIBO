### 问题清单

| 编号 | 优先级 | 问题描述 | 建议修复方案 |
|------|--------|----------|------------|
| 001 | P0 | **异步代码中存在阻塞 IO**：`GraphDBClient.query_rules` 方法定义为 `async`，但内部使用同步库 `requests` 进行 HTTP 调用。这将阻塞事件循环，导致高并发下查询引擎吞吐量急剧下降。 | 替换为异步 HTTP 客户端（如 `aiohttp` 或 `httpx` 的 async 模式），确保所有 IO 操作非阻塞。 |
| 002 | P0 | **Celery 与 Asyncio 混合架构风险**：`CompileWorker` 在 Celery 任务中使用 `asyncio.run()`，同时 Worker 池配置为 `gevent`。在特定版本组合下，`gevent` 会 patch 标准库，可能与 `asyncio` 事件循环冲突，导致死锁或协程无法调度。 | 建议统一模型：要么 Celery 任务全同步（移除 `async/await`），要么使用支持原生 Async 的任务队列（如 Celery 5.0+ 配合 `asyncio` 池，或改用 RQ/ARQ）。避免在 Gevent 池中混用 `asyncio.run`。 |
| 003 | P1 | **核心逻辑缺失：RDF 到 SQL 的映射机制**：文档提到从 GraphDB (RDF) 规则生成 SQL 模板，但未说明具体的映射算法。RDF 的图结构关系（多跳、继承）如何转换为关系型 SQL 的 JOIN？这是系统的核心复杂度，目前设计过于简化。 | 补充“语义映射层”设计文档。详细说明如何将 FIBO 本体属性映射到物理表字段，如何处理多表关联、继承关系的 SQL 生成逻辑，并提供原型验证。 |
| 004 | P1 | **L0 规则更新的一致性风险**：L0 变更（全局标准）采用“懒加载 + 后台编译”策略，查询时允许继续使用旧版本。若 L0 涉及监管合规或核心财务逻辑变更，旧版本数据可能导致合规风险或数据错误。 | 引入“灰度发布”或“强制生效时间”机制。对于标记为 Critical 的 L0 变更，应阻断查询或强制同步编译，而非允许使用过期规则。需明确业务容忍度。 |
| 005 | P2 | **WASM 沙箱过度设计**：DSL 执行强制要求 WASM 沙箱。对于简单的算术公式，WASM 引入额外的编译开销、部署复杂度及调试难度。若仅为安全，Python 的 `ast` 限制 + 沙箱进程可能更轻量。 | 评估安全需求等级。若仅为防止代码注入，建议使用受限的 Python 解释器（如 `restrictedpython`）或纯表达式引擎（如 `simpleeval`）。若坚持 WASM，需补充性能基准测试（编译 + 执行耗时）。 |
| 006 | P2 | **LLM Fallback 成本与延迟失控**：意图分类置信度 <0.7 时直接调用 LLM。在高并发场景下，LLM 调用延迟（秒级）和 Token 成本将不可控，且可能成为系统瓶颈。 | 增加 LLM 调用的限流器（Rate Limiter）和预算控制。对于低置信度查询，优先引导用户使用“标准问法”或人工客服，仅对高价值租户开放 LLM 解析。 |
| 007 | P2 | **缓存一致性隐患**：编译结果存入 Redis，但 `CompileWorker` 失败重试时，若部分租户成功部分失败，Redis 中可能残留不一致的版本状态。且未提及 Redis 持久化策略，重启可能导致编译状态丢失。 | 引入“编译状态机”存储在 PostgreSQL 中作为 Source of Truth，Redis 仅作为热点缓存。确保编译任务的原子性，失败时回滚状态标记。 |
| 008 | P3 | **内部服务鉴权缺失**：架构图中 API Gateway 负责鉴权，但服务层内部（如 Query Engine 调用 Compiler）缺乏 mTLS 或 Service Account 鉴权。若内网被突破，服务间调用无防护。 | 补充服务间通信安全设计，建议实施 mTLS 或基于 JWT 的内部服务鉴权机制。 |
| 009 | P3 | **可观测性不足**：监控指标中缺少分布式追踪（Trace ID 透传）。在异步、多服务（Gateway->Engine->Compiler->DB）架构中，缺乏全链路追踪将导致故障定位困难。 | 集成 OpenTelemetry 或 SkyWalking，确保 `trace_id` 在所有异步上下文和 Celery 任务中透传。 |
| 010 | P4 | **SQL 生成白名单维护困难**：`SQLGenerator` 中硬编码了 `ALLOWED_TABLES` 和 `ALLOWED_COLUMNS`。随着业务表结构变更，需频繁修改代码并重新部署。 | 将白名单配置化，存入 PostgreSQL 元数据表，支持动态加载和热更新，减少代码耦合。 |

### 评审结论：有条件通过

**理由：**
1.  **架构整体可行**：分层架构清晰，技术选型（GraphDB + PG + Redis）符合场景需求，规则引擎与查询引擎分离设计合理。
2.  **存在阻塞性技术风险**：**P0 级别的异步 IO 阻塞**和**Celery/Async 混合模型冲突**是严重的技术债务，若不修复将直接导致生产环境性能崩塌或不稳定。
3.  **核心逻辑待完善**：**RDF 到 SQL 的映射**是本项目最大的技术难点，当前文档对此描述过于简略，存在交付风险。
4.  **业务一致性需确认**：L0 规则更新的“最终一致性”策略需业务方确认是否接受合规风险。

**后续行动建议：**
1.  **立即修复 P0 问题**：重构 `GraphDBClient` 为异步 IO，重新评估 Celery 异步任务模型。
2.  **补充核心设计**：在下一版本中增加“语义映射层”的详细设计文档及原型验证结果。
3.  **明确业务边界**：与业务方确认 L0 规则更新的 SLA 要求，调整编译策略。
4.  **安全与成本评估**：对 WASM 方案进行 POC 验证，对 LLM 调用进行成本测算。
